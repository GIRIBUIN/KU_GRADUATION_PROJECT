# 백엔드 구현 기준

## 1. 목적

UI 목업과 실제 앱 로직 사이의 기준을 정한다.

현재 앱은 Drift 기반 로컬 DB, Repository, Service를 실제 화면에 연결해 사용한다.
문서는 UI 목업이 아니라 현재 데이터 흐름과 저장 구조를 기준으로 유지한다.

## 2. 레이어 구분

### Models

앱에서 저장하고 처리할 순수 데이터 구조를 정의한다.

- Task
- Tag
- Folder
- NotificationSetting
- AppSettings
- EcampusSyncMetadata
- ParsedEcampusTask
- SyncResult

UI 표시용 색상, 라벨, 배지 문구는 가능하면 모델에 넣지 않는다.

### Repositories

앱의 상위 레이어가 데이터를 읽고 쓰는 진입점이다.

- 로컬 DB 구현이 바뀌어도 Repository 인터페이스는 유지한다.
- UI는 가능하면 DB나 service를 직접 호출하지 않는다.
- 현재 구현은 Drift 기반 repository를 사용한다.

### Services

외부 시스템, 파싱, 동기화 같은 작업을 담당한다.

- e-campus 로그인
- todo HTML 요청
- HTML 파싱
- 동기화 분류
- 알림 예약

## 3. UI와 백엔드 연결 원칙

현재 연결 흐름:

```text
Repository/Service
→ 도메인 모델
→ UI 표시용 ViewModel 변환
→ 화면 출력
```

UI에서 필요한 `dueLabel`, 색상, 상태 배지 문구는 ViewModel 또는 helper에서 만든다.

## 4. 구현 우선순위

1. Drift DB 스키마와 마이그레이션 유지
2. Repository 인터페이스와 Drift 구현 유지
3. e-campus WebView 로그인 세션 처리
4. todo HTML 요청 및 파싱
5. sourceKey 생성과 동기화 분류
6. 동기화 결과 미리보기/반영
7. 로컬 알림 예약과 재예약

## 5. 주의사항

- e-campus 계정 정보는 로그에 남기지 않는다.
- 현재 구현은 아이디/비밀번호를 저장하지 않는다.
- HTML 파싱 실패로 앱이 중단되면 안 된다.
- 삭제, 완료, 제외 상태는 서로 다른 의미로 관리한다.
- 현재 구현은 e-campus 원본에서 사라진 active task 중 마감 전 항목을 완료 처리한다.
