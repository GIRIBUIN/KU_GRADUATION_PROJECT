# 기능 설계

## 1. 서비스 목적

건국대학교 학생이 `e-campus 체크리스트 ToDo`와 `개인 ToDo`를 하나의 앱에서 통합 관리할 수 있도록 하는 모바일 앱이다.

e-campus에서 가져오는 대상은 학교 행사나 일반 캘린더 일정이 아니라, 온라인 강의, 과제, 팀프로젝트 과제처럼 사용자가 실제로 처리해야 하는 학업 체크리스트 항목이다. 사용자는 이를 개인 작업과 함께 보고, 태그, 우선순위, 알림, 폴더를 이용해 관리한다.

## 2. 문제 정의

- e-campus에 흩어진 학업 ToDo와 개인 ToDo를 따로 확인해야 한다.
- 과제, 강의, 팀플 마감일을 놓치기 쉽다.
- 학교 작업과 개인 작업의 우선순위를 한 기준으로 관리하기 어렵다.
- 사용자가 직접 정리한 태그나 폴더 기준으로 학업 작업을 재분류하기 어렵다.

## 3. 핵심 기능

### 3.1 Task 관리

- 개인 task 생성, 조회, 수정, 삭제
- e-campus에서 가져온 task 조회, 수정, 삭제
- 완료/미완료 전환
- parent/sub task 관계 지원
- 태그, 우선순위, 알림, 폴더 설정

e-campus task도 앱 안에서는 일반 task처럼 수정할 수 있다. 단, 동기화 판단을 위해 e-campus 고유 식별값은 별도로 보관한다.

- `Task` 모델은 `memo`, `sortOrder` 필드를 가지며, e-campus 관련 메타데이터(`sourceKey`, `sourceTitle`, `sourceDueDate`, `sourceCourse`, `sourceType`, `lastSyncedAt`)를 저장합니다.

### 3.2 e-campus 동기화

- 사용자가 동기화 아이콘을 눌러 e-campus 체크리스트를 가져온다.
- 저장된 계정 정보와 자동 동기화 설정이 있는 경우 앱 실행 시 동기화를 시도할 수 있다.
- 동기화 결과는 `가져올 항목`과 `가져오지 않을 항목`으로 나누어 보여준다.
- 신규 항목과 업데이트 후보는 사용자가 선택한 경우에만 앱 task로 반영한다.

### 3.3 홈

홈 화면은 전체 목록보다 `오늘 할 일`과 `마감 임박` task를 중심으로 구성한다.

- 오늘 할 일
- 마감 임박
- 미완료 요약
- 마지막 동기화 상태
- 개인 task 빠른 추가

### 3.4 태그

- task는 여러 태그를 가질 수 있다.
 - task는 여러 태그를 가질 수 있다.
 - 태그는 이름과 색상만을 가진다. (현재 코드상 `Tag` 모델에는 기본 우선순위 필드가 없습니다.)
 - task 우선순위가 없는 경우 태그 기반 기본 우선순위를 사용하는 설계가 문서에는 정의되어 있으나, 현재 코드에서는 태그의 기본 우선순위가 구현되어 있지 않으므로 우선순위 산정은 태그 기반 항목이 적용되지 않습니다.

### 3.5 폴더

폴더는 태그와 별개의 사용자 지정 task 모음이다.

- task는 여러 폴더에 들어갈 수 있다.
- 폴더에 task를 직접 추가하거나 제거할 수 있다.
- 태그별 task 목록에서 선택 추가 또는 일괄 추가할 수 있다.

추가 구현(코드 기준): 폴더는 중첩 구조(`parentFolderId`)를 지원하고, `icon`, `color`, `sortOrder` 같은 메타데이터를 가집니다. 정렬 순서(`sortOrder`)를 통해 폴더의 표시 순서를 제어합니다.

### 3.6 알림

- task별 알림 설정을 지원한다.
- task 하나에는 알림 설정 하나만 둔다.
 - task별 알림 설정을 지원한다.
 - task 하나에는 알림 설정 하나만 둔다.
 - 알림 기준은 `마감일 기준 N분 전 + 특정 시간`으로 동작합니다. (현재 코드에서 `NotificationSetting.daysBeforeDue`는 분(minutes) 단위로 해석되어 예약 시 `Duration(minutes: daysBeforeDue)`로 계산됩니다.)
 - `notifyTime`(문자열)와 예약된 시각(`scheduledAt`)이 DB에 저장됩니다. 로컬 알림은 Flutter Local Notifications 플러그인으로 스케줄링됩니다.
 - 개인 task와 e-campus task 모두 알림을 설정할 수 있다.

### 3.7 위젯

위젯은 가능하면 구현하는 기능으로 고려하고 있다. 서비스 기능이 구현 우선순위가 높다.

- 조회형 위젯으로 설계한다.
- 오늘 할 일과 마감 임박 task 중심으로 보여준다.
- 표시 개수는 최대 3개 정도로 단순하게 둔다.
- 위젯에서 직접 완료 체크하는 기능은 후순위로 둔다.

## 4. Task 상태

| 상태 | 의미 | 적용 대상 |
| --- | --- | --- |
| `active` | 진행 중인 task | 개인, e-campus |
| `completed` | 완료 처리된 task | 개인, e-campus |
| `deleted` | 삭제되었지만 복구 가능한 task | 개인, e-campus |
| `excluded` | e-campus에서 발견되었지만 가져오지 않는 항목 | e-campus |

`excluded`는 e-campus task에만 사용한다.

참고(코드 기준): 코드 상의 편의성을 위해 앱 전반에서 `isArchived` 판정은 `completed`, `deleted`, `excluded` 상태를 모두 아카이브로 간주하는 로직이 존재합니다. (`Task.isArchived`가 이 세 상태를 포함함)

## 5. 삭제 정책

### 개인 task

- 삭제 시 목록에서는 숨긴다.
- 삭제 후 7일 동안 복구 가능하다.(설정에서 기간 조정 추가 구현)
- 7일 이후 완전 삭제할 수 있다.

### e-campus task

- 삭제 또는 제외 시 목록에서는 숨긴다.
- `sourceKey`와 `sourceDueDate`는 유지한다.
- 다음 동기화 때 같은 항목은 신규가 아니라 `가져오지 않을 항목`으로 보여준다.
- `sourceDueDate` 이후 삭제/제외 기록을 정리할 수 있다.
- `sourceDueDate`가 없으면 삭제 시점 기준 7일 동안 유지한다.

## 6. 우선순위 정책

우선순위 적용 순서는 다음과 같다.

1. task에 직접 설정한 우선순위
2. task 우선순위가 없으면 태그 기본 우선순위(현재 코드에서는 태그의 기본 우선순위 필드가 구현되어 있지 않아 이 단계는 적용되지 않습니다)
3. 태그 기본 우선순위도 없으면 앱 기본 우선순위

task에 여러 태그가 있고 task 우선순위가 없는 경우, 태그 중 가장 높은 기본 우선순위를 사용한다.

## 7. Parent/Sub Task 정책

- task는 parent/sub 관계를 가질 수 있다.
- sub task는 제목과 완료 여부를 가진다.
- 홈에서는 parent task만 표시하고 sub task는 진행률로 보여준다.
- 목록에서는 parent task를 펼쳐 sub task를 확인할 수 있다.
- 상세/수정 화면에서 sub task 추가, 수정, 삭제, 완료 처리를 한다.
- sub task가 모두 완료되어도 parent task를 자동 완료하지 않는다.

**데이터베이스/저장 필드(요약, 코드 기준)**

- `Task` 주요 필드: `id`, `origin`, `status`, `title`, `dueDate`, `priority`, `memo`, `parentTaskId`, `tagIds`, `folderIds`, `sortOrder`, `createdAt`, `updatedAt`, `completedAt`, `deletedAt`, `ecampus`(sourceKey, sourceTitle, sourceDueDate, sourceCourse, sourceType, lastSyncedAt)
- `NotificationSetting` 주요 필드: `id`, `taskId`, `enabled`, `daysBeforeDue`(분 단위), `notifyTime`(문자열), `scheduledAt`
- `Tag` 주요 필드: `id`, `name`, `color` (현재 태그 우선순위 필드 없음)
- `Folder` 주요 필드: `id`, `name`, `color`, `icon`, `parentFolderId`, `sortOrder`

위 항목들은 현재 코드베이스(`lib/data/models/`, `lib/data/local/app_database.dart`)에 따라 문서화한 내용입니다.
