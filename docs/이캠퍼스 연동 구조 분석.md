# e-campus 연동 구조 분석

## 1. 연동 목적

e-campus 체크리스트에 있는 온라인 강의, 과제, 팀프로젝트 과제 등의 학업 ToDo를 가져와 앱의 task로 관리하기 위해 연동한다.

## 2. 기본 원칙

- e-campus 데이터는 읽기 전용으로 수집한다.
- 사용자의 제출, 수정, 삭제를 e-campus에 반영하지 않는다.
- 앱은 e-campus 원본 시스템을 변경하지 않는다.
- 동기화는 필요한 순간에만 수행한다.

## 3. 로그인 방식

### 계정 정보 저장을 하지 않는 경우

```text
동기화 요청
→ 사용자 로그인 입력
→ e-campus 체크리스트 수집
→ 로그아웃 및 세션 정리
```

### 계정 정보 저장에 동의한 경우

```text
동기화 요청 또는 앱 실행 시 자동 동기화
→ 기기 내 보안 저장소에서 계정 정보 조회
→ 로그인 시도
→ e-campus 체크리스트 수집
→ 로그아웃 및 세션 정리
```

## 4. 자동 동기화

- 기본 동기화는 수동 동기화이다.
- 사용자가 동의한 경우 앱 실행 시 1회 자동 동기화를 시도한다.(아직 고민 중)
- 새 항목이 없으면 가볍게 안내한다.
- 새 항목 또는 업데이트 후보가 있으면 사용자에게 감지 사실을 알리고, 사용자가 확인하면 동기화 결과 화면을 보여준다.

## 5. 동기화 결과 분류

### 가져올 항목

- DB에 없는 신규 e-campus 항목
- `sourceKey`는 같지만 원본 정보가 달라진 업데이트 후보

### 가져오지 않을 항목

- 이미 가져온 항목
- 완료된 항목
- 삭제된 항목
- 제외된 항목

## 6. 업데이트 후보

`sourceKey`가 같고 다음 원본 정보가 달라진 경우 업데이트 후보로 본다.

- `sourceTitle`
- `sourceDueDate`
- `sourceCourse`
- `sourceType`

업데이트 후보는 사용자가 선택했을 때만 task 값을 원본 기준으로 덮어쓴다.

## 7. 원본에서 사라진 항목

e-campus 체크리스트에서 항목이 사라져도 앱에서 자동 완료 처리하지 않는다.

이유:

- 사용자가 제출해서 사라진 것인지 확인할 수 없다.
- 교수자가 삭제했는지 확인할 수 없다.
- 기한 만료로 사라진 것인지 확인할 수 없다.

따라서 앱에서는 기존 task 상태를 유지하고, 사용자가 직접 완료 또는 삭제 처리한다.

## 8. 동기화 실패 케이스

- 로그인 실패
- 비밀번호 변경
- 네트워크 오류
- e-campus 서버 오류
- HTML 구조 변경
- 세션 만료

실패 시 사용자에게 원인을 구분할 수 있는 메시지를 제공한다.

## 9. 보안 처리

- 계정 정보 저장은 사용자 동의 기반이다.
- 계정 정보는 기기 내 보안 저장소에만 저장한다.
- 자체 서버에 계정 정보를 저장하지 않는다.
- 동기화 완료 후 로그아웃한다.
- 세션과 쿠키를 정리한다.
- 로그에는 계정 정보와 세션 값을 남기지 않는다.
